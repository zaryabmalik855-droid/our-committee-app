# Responsible AI & Ethics — Our Committee AI

## Overview

Our Committee AI handles sensitive financial data for Pakistani women and small-scale vendors. Responsible AI is not optional — it is a core design principle. This document covers our hallucination mitigations, bias assessment, privacy review, content moderation, limitations, and AI disclosure.

---

## 1. Hallucination Risks & Mitigations

### Risk: AI invents financial figures
**Severity**: HIGH — a wrong balance or loan amount could lead to financial decisions based on false data.

**Mitigations implemented**:
| Mitigation | Location |
|---|---|
| RAG grounding — all financial answers cite real data | `rag_service.py` |
| Negative prompting — "Never hallucinate financial figures not in the context" | `prompts.yaml` |
| Tool-first architecture — agent runs DB query before LLM generates | `agent_service.py` |
| Evaluation — keyword coverage checks (must cite real values) | `eval/run_evaluation.py` |
| Context injection — full user data in every prompt | `llm_service.py` |

### Risk: AI fabricates loan approval
**Severity**: CRITICAL — wrongful approval/rejection signals could cause real financial harm.

**Mitigation**: System prompt explicitly states AI cannot approve/reject loans; it must defer to the Manager. Negative constraint: "Do NOT approve or reject loans on your own."

## 2. Bias Assessment

### Potential Biases Identified

| Bias Type | Risk | Status |
|---|---|---|
| **Language bias** | App may perform better in English than Urdu | ⚠️ Monitored — 5/23 eval cases are Urdu |
| **Gender bias** | LLM training data may have biased financial advice | ⚠️ Mitigated — answers are data-driven, not opinion-driven |
| **Loan risk bias** | Risk algorithm may disadvantage users with no payment history | ⚠️ Noted — baseline score is 0.5 (neutral) |
| **Class bias** | Model may not understand informal Pakistani savings terms | ✅ Mitigated — system prompt explains Kameti/Bisi context |

### Sample Output Review
Manual review of 10 sample outputs confirmed:
- No gendered assumptions in loan advice
- No bias toward English-speaking users (Urdu responses equally detailed)
- Loan risk scores correlate with payment data, not demographics

## 3. Data Privacy Review

### What Data Is Processed
| Data | Sent to External API? | Notes |
|---|---|---|
| User name | ✅ Yes (Gemini/OpenAI) | First name only — not CNIC or ID |
| Email address | ✅ Yes | Used as user identifier |
| Balance amount | ✅ Yes | Financial figure in prompt |
| Committee details | ✅ Yes | Aggregate financial data |
| Payment receipts | ✅ Yes | Transaction history |

### What We Do NOT Process
- ❌ CNIC / National ID numbers
- ❌ Bank account numbers or IBAN
- ❌ Phone numbers
- ❌ Physical address
- ❌ Biometric data

### Privacy Safeguards
1. **API key security**: Keys stored in `.env` (not hardcoded), excluded from git via `.gitignore`
2. **User scoping**: PGVector retrieval filtered by `user_email` — users cannot access other users' data
3. **No logging of sensitive content**: API logs record endpoints and latency, not message content
4. **Data minimization**: Only data necessary for the query is injected into the prompt
5. **No data persistence in LLM**: Gemini/OpenAI APIs do not retain user data for training (as per their ToS)

### Consent
> **AI Disclosure (shown in UI)**: "⚠️ AI Disclosure: Insights, loan analysis, and drafts are generated by AI. Please review and verify all financial figures independently."

Users are informed they are interacting with AI at every session.

## 4. Content Moderation Strategy

Our moderation runs in **two layers**:

### Layer 1: Rule-Based (Backend — `llm_service.py`)
- **Prompt injection blocklist**: 14 patterns including "ignore previous instructions", "jailbreak", "act as", "DAN mode"
- **Keyword blocklist**: Offensive terms in English and Urdu (abuse, harassment, violence terms, Pakistani slang insults)
- **Always runs first** — before any API call

### Layer 2: API-Based (Optional)
- **OpenAI Moderation API**: When `LLM_PROVIDER=openai`, every message is checked via `client.moderations.create()`
- **Gemini safety filters**: Gemini has built-in HARM_CATEGORY safety settings active by default

### Fallback Response
When moderation blocks a message:
```
"I cannot fulfill this request as it violates safety guidelines."
```

## 5. Limitations

> **Important**: These limitations must be communicated to users and in any technical report.

1. **No real-time data**: The app works with data sent by the client app — it does not connect to live bank APIs.
2. **Offline fallback degrades quality**: Without an API key, FakeEmbeddings are used — RAG retrieval becomes random.
3. **Urdu quality varies**: Complex financial Urdu may be handled less precisely than English; review all Urdu outputs.
4. **Loan risk is heuristic**: The confidence score is based on payment history count, not a trained ML model.
5. **Rate limiting**: Free tier Gemini API limits (15 RPM / 1,500 RPD) mean heavy concurrent use will hit quotas.
6. **No persistent conversation memory**: Chat history is held client-side and lost on page refresh.
7. **PGVector requires PostgreSQL**: The app cannot run in full RAG mode without a PostgreSQL + pgvector server.
8. **Demo data only**: Streamlit frontend uses mock data — real integration requires the Flutter app.
